# Configuration

Config Sheet Forge has two local files:

- `.config-sheet-forge/config.json`
- `.config-sheet-forge/registry.json`

These files are generated by `config-sheet-forge init` and are intentionally ignored by git.

## config.json

```json
{
  "schemaVersion": "1",
  "provider": "lark",
  "rootUrl": "",
  "rootToken": "",
  "rootObjectType": "",
  "registryPath": ".config-sheet-forge/registry.json",
  "cacheDirectory": ".config-sheet-forge/cache",
  "providerSettings": {
    "larkCliPath": "lark-cli"
  }
}
```

Use `rootUrl` for a human-checkable URL. Use `rootToken` only when a provider cannot work from a URL. Neither field should be committed when it points to a real tenant document.

## registry.json

```json
{
  "schemaVersion": "1",
  "tables": [
    {
      "id": "items",
      "name": "Items",
      "provider": "lark",
      "spreadsheet": "",
      "sheetId": "",
      "range": "A1:Z500",
      "localSourcePath": ""
    }
  ]
}
```

`id` is the stable machine id used by the CLI and CI. `name` is the human label shown in reports. `spreadsheet`, `sheetId`, and `range` are provider-specific values read from config.

## lark-cli Path

By default the Lark provider uses `lark-cli` and resolves it from the local environment. If the executable is installed in a non-standard location, set either:

- `providerSettings.larkCliPath` in `.config-sheet-forge/config.json`
- `LARK_CLI_PATH` in the environment

On Windows, the provider resolves npm shims such as `lark-cli.cmd` explicitly instead of relying on shell behavior.

## Secrets

Do not store app secrets, access tokens, or refresh tokens in these files. The Lark provider expects `lark-cli` to manage local auth outside the repository.

## Root Discovery

`discover-root` can search and rank candidates, but it never writes a root automatically. A wrong root can sync the wrong source of truth, so choosing one is always a human action.
